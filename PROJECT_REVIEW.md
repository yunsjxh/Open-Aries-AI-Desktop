# Open-Aries-AI 项目评估与改进建议（2026-04-11）

## 1) 项目现状速览

这个项目是一个 **Windows + C++17 单体可执行程序**，核心闭环是：

1. 屏幕截图；
2. 调用 OpenAI 兼容接口做视觉/文本推理；
3. 解析 AI 返回动作；
4. 执行动作（鼠标、键盘、应用、文件、命令）；
5. 回传结果并继续下一轮。

从 README 能看到：
- 已支持 OpenAI 兼容 API、多 Provider、视觉、多文件操作、更新检查。  
- 当前强调“可直接运行、功能覆盖广”的工程目标。  

整体判断：**功能原型已较完整，工程化与安全边界是下一阶段关键。**

---

## 2) 主要优点

- **模块化拆分较清晰**：Provider、ActionParser、ActionExecutor、FileManager、SecureStorage 分层明显，便于后续替换实现。
- **中文场景支持意识强**：提示词、日志、命令交互、Unicode 路径处理都有针对性设计。
- **能力面广**：不仅是 GUI 自动化，还集成文件与命令执行，具备“桌面代理”雏形。

---

## 3) 风险与短板（按优先级）

### P0（应优先）

1. **安全模型偏弱：明文逻辑可逆 + 规则黑名单可绕过**  
   当前密钥保护主要基于硬件指纹 + XOR/混淆，不属于强密码学保护；`Execute` 的 PowerShell 防护主要依赖字符串黑名单。对于本地恶意样本和提示词注入场景，风险较高。

2. **动作执行权限过大，缺少“能力白名单 + 显式授权”机制**  
   目前文件改写、执行命令、运行可执行文件等能力都可被模型间接触发，若模型误判或遭污染，潜在破坏面大。

3. **解析与 JSON 处理偏“手搓字符串”**  
   包含手写 JSON 字段提取、手写 action 解析。长期维护中容易在边界输入、转义字符、异常格式上出错。

### P1（建议尽快）

4. **可测试性不足**  
   目前几乎是头文件内联逻辑 + 主流程耦合，单元测试入口不明显。动作解析、命令安全过滤、文件编码读写都很适合做测试集。

5. **可观测性仍偏基础**  
   现在以文本日志为主，缺少结构化日志字段（trace_id / round_id / action_id / duration / token usage），不利于定位线上问题。

6. **平台耦合强**  
   强依赖 Win32/GDI+/WinINet/WinHTTP，短期是合理取舍，但建议抽象平台层，减少未来迁移成本（如 Linux agent / remote executor）。

### P2（中期优化）

7. **依赖管理与构建体系较简陋**  
   README 给出命令行编译，但缺少 CMake、CI、发布校验矩阵；版本演进后维护成本会明显上升。

8. **文档缺“安全使用边界”**  
   当前文档偏能力展示，建议补充风险声明、最小权限运行建议、企业环境禁用项。

---

## 4) 建议路线图（可执行）

### 4.1 两周内（安全与稳定先行）

- 引入 **能力开关（Capability Flags）**：默认关闭高危能力（Execute / FileWrite / FileDelete / FileRun），用户显式开启。
- 给高危动作增加 **二次确认**：例如“将要执行 PowerShell，是否继续”。
- `SecureStorage` 切换到 **Windows DPAPI**（`CryptProtectData` / `CryptUnprotectData`）为主方案。
- 把 Action 输出改成 **严格 JSON schema**，并在解析前做 schema 校验。

### 4.2 一个月内（工程化）

- 建立 CMake + GitHub Actions（编译、lint、基础测试）。
- 为 `ActionParser`、`isCommandAllowed`、`FileManager` 建立回归测试用例。
- 增加结构化日志（JSON Lines），并加入每轮动作的耗时统计。

### 4.3 两个月内（产品化）

- 设计“**安全模式 / 开发者模式**”双配置。
- 增加“干跑（dry-run）模式”：只生成动作不执行，便于审阅。
- 加入 Provider 超时、重试、熔断与更细粒度错误码。

---

## 5) 结论

这是一个完成度不错的桌面智能代理原型，尤其在中文和实用功能覆盖上有明显优势。  
下一阶段最关键的是从“能用”走向“可控且可信”：**先补安全边界，再补测试与工程化。**
